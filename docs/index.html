Backups for Lightning & RGB
that pay for themselves

PIVSS is an incentivized storage market: providers earn sats for holding your encrypted wallet state — but only for as long as they can prove it's still there. No trusted single vendor. No blind trust in a payment either.

A seed phrase doesn't save everything

On-chain Bitcoin is forgiving — lose your device, restore twelve words, get everything back. Lightning and RGB are not.

⚡ Lightning channel state

Every payment changes your channel's commitment state. Restore an old backup and you risk broadcasting a revoked commitment — Lightning's penalty mechanism hands your balance to the other side. Static channel backups only get you a cooperative force-close; in-flight funds are still gone.

🎨 RGB client-side assets

RGB keeps asset data in your device's local stash — there's no global chain to replay it from. Lose the data, lose the asset. This is the single biggest UX blocker between RGB and mainstream custody.

The incentive loop

Storage providers publish a price and a real Lightning offer on nostr. Clients pay for provable custody, one billing period at a time.

1

Provider advertises

A signed nostr event (kind 38831): endpoint, price in sats/MiB, and a real BOLT12 offer.

2

Client uploads

An opaque, already-encrypted blob. Re-uploading creates a new version — full history is kept.

3

Client challenges

A fresh nonce + random byte ranges. The provider must read the actual bytes to answer — no caching around it.

4

Proof → payment

Valid proof, real BOLT12 payment. Failed proof, payment withheld. Dishonesty stops earning immediately.

Trust model

The provider is zero-knowledge. Every payload is treated as an opaque, already-encrypted blob — bring your own ciphertext, or use the built-in AES-256-GCM/Argon2id helper. The provider, and anyone who compromises it, sees only ciphertext, size, and access timing. Never your channel state or your assets.

Payment is real, not a promise. Both sides run an embedded breez-sdk-liquid wallet. The provider only ever records a payment its own wallet actually observed — a client asserting "I paid" is rejected outright once a real wallet is connected.

Availability isn't fully trustless yet. A single provider can still go offline — it just can't lie about holding your data, or read it. Spreading backups across independent providers (erasure coding) is the next milestone toward that.

Under the hood

Four small Rust crates, no bloat.

pivss-coreVSS protobuf types (hand-written, no protoc needed), the AES-256-GCM/Argon2id envelope, BitTorrent v1 metainfo + magnet links, proof-of-storage challenges, minimal NIP-01 nostr signing.
pivss-lnThin wrapper around breez-sdk-liquid: durable BOLT12 offer creation, paying an offer with a correlating payer note, streaming confirmed incoming payments.
pivss-serverThe provider: axum HTTP API, pluggable storage (in-memory or a real LDK vss-server), BitTorrent seeding via carl, nostr announcements, embedded host panel + web client.
pivss-clientThe wallet side: API client library and a CLI — backup, verify, watch, pay, and discover (find providers via nostr).

Try it in one command

Docker Compose, zero config, no API key needed for the demo.

# clone, then:
$ docker compose up -d server

Open http://localhost:8339/app — a complete zero-install web client: pick a file, upload it, verify proof-of-storage, pay, and discover other providers, all from the browser. http://localhost:8339/panel is the provider's own view.

Prefer the CLI? It's dockerized too — docker compose run --rm client backup /uploads/yourfile --kind lightning. Full instructions in the README.

Where this actually stands

Built fast, but we're not going to oversell it — here's what's real and what's still a roadmap item.

Live and verified

  • Real BOLT12 payments (mainnet + regtest) via breez-sdk-liquid, both sides
  • Provider never trusts a client's payment claim — only its own wallet
  • Proof-of-storage: honest / tampered / replayed-nonce, all tested
  • Client-side encryption, zero-knowledge server
  • Nostr publish and discovery, both signature-verified round-trip
  • Docker Compose — full loop tested end-to-end through containers

Still on the roadmap

  • Multi-provider erasure coding — the real availability guarantee
  • Anti-outsourcing proofs (a dishonest provider can still proxy a challenge)
  • Server-side auth / rate-limiting
  • Merkle-commitment proofs for thin clients that don't hold the file
  • Non-repudiable payment proofs (pending rust-lightning #4297)
